The Problem

The NCTU Plus frontend is a React SPA for NYCU students to browse and post course reviews. Authentication is entirely backend-owned: the Go API handles the NYCU OAuth redirect and sets an httpOnly JWT cookie. The frontend never touches tokens directly — it can only probe session state by calling the API.

This shapes every auth-related decision: there is no login form, no token storage, no logout handler on the client side. The frontend’s job is to react to what the backend tells it.

State Architecture

Selective Persistence

The Redux store holds eight slices. Only two are persisted to localStorage:

SlicePersistedReason
saveFormYesDraft comment survives navigation and tab close
saveNewCourseYesCourse creation draft survives navigation
loginNoRe-hydrated from API on every mount
id, nameNoRe-hydrated from API on every mount
windowWidthNoRe-computed from window.innerWidth
searchKeywordNoTransient navigation state
editIDNoTransient edit-in-progress marker

redux-persist wraps only the two draft slices rather than the root reducer. This avoids stale session state from a previous browser session being treated as live auth on the next visit.

const saveFormPersistConfig = { key: 'saveForm', storage };
const saveFormPersistedReducer = persistReducer(saveFormPersistConfig, saveFormReducer);

const rootReducer = combineReducers({
  saveForm: saveFormPersistedReducer,
  saveNewCourse: saveNewCoursePersistedReducer,
  login: loginReducer,
  id: idReducer,
  // ...
});

Implicit Session Hydration

There is no explicit “check auth” route or loading state. App.js fires a POST /oauth/me on mount. Success populates id, name, and login in the store; any failure (network error, expired cookie, 401) silently sets logged-out state. Components that require auth check selectLogin from the store and gate their UI accordingly.

const getUserInfo = async () => {
  try {
    const response = await fetch(REACT_APP_API_URL + '/oauth/me', {
      method: 'POST',
      credentials: 'include'
    }).then(res => res.json());
    dispatch(updateID(response.student_id));
    dispatch(updateName(response.name));
    dispatch(updateLogin(true));
  } catch {
    dispatch(updateID(''));
    dispatch(updateName(''));
    dispatch(updateLogin(false));
  }
};

useEffect(() => { getUserInfo(); }, []);

credentials: 'include' is required so the browser sends the httpOnly cookie cross-origin. Without it, the cookie is silently omitted and every request looks unauthenticated.

Pagination Without a Count Endpoint

The comment list supports pagination. The API has a GET /comments/pages endpoint that returns { maxPages }. It wasn’t reliable at the time the pagination UI shipped, so the fallback walks pages sequentially until it finds one with fewer than 30 results:

/* Ideal way */
const getMaxPage = async () => {
  const response = await fetch(max_page_api_url).then(res => res.json());
  setMaxPage(response.maxPages);
};

/* Now we are using this */
const uglyGetMaxPage = async () => {
  let page = 1;
  while (true) {
    const response = await fetch(`${search_api_url}?q=${searchKey}&page=${page - 1}`)
      .then(res => res.json());
    if (response.length < 30) {
      if (response.length > 0) setMaxPage(page);
      else setMaxPage(page - 1);
      break;
    }
    page++;
  }
};

The comments in the code name this what it is. The fix is already known — it’s waiting on the backend endpoint to be stable.

Debounced Course Search

The new-comment form lets users search for a course to attach their review to. A naive onChange → fetch on every keystroke would hammer the search API. The implementation uses a ref-based timer to cancel the pending request when the next keystroke arrives:

const typingTimeoutRef = useRef(null);

useEffect(() => {
  const controller = new AbortController();
  const delaySearch = () => {
    if (typingTimeoutRef.current) {
      clearTimeout(typingTimeoutRef.current);
    }
    typingTimeoutRef.current = setTimeout(() => {
      if (searchQuery.trim() !== '') {
        searchCourse(searchQuery);
      } else {
        setSuggestions([]);
      }
    }, 300);
  };

  delaySearch();

  return () => {
    controller.abort();
    if (typingTimeoutRef.current) clearTimeout(typingTimeoutRef.current);
  };
}, [searchQuery]);

Using a ref rather than useState for the timer ID avoids triggering a re-render when the timer is set or cleared — the value is only ever used by cleanup logic, not by the render output.

Responsive Layout via Redux

Window width is stored in the Redux store and updated on every resize event. Components read selectWindowWidth and apply inline styles directly in JSX:

<h1 style={{
  fontSize: windowWidth > 1200 ? '40px' : windowWidth > 600 ? '30px' : '20px',
  marginBottom: windowWidth > 1200 ? '80px' : '30px',
}}>
  借鏡學長姐的經驗，成為交大選課大師
</h1>

This is more verbose than CSS media queries but lets the same breakpoint values drive both layout and conditional rendering of JSX subtrees — useful when entire components, not just styles, need to swap at a breakpoint.

The useLayoutEffect is intentional: it runs synchronously before the browser paints, so the initial render already has the correct window width rather than flickering from a default value.

Takeaway

The draft slice is the right thing to persist. Session state belongs to the server; persisting it client-side creates a split-brain problem. Form drafts belong to the user; losing them on navigation is a usability bug. redux-persist applied selectively solves both.

Name the workaround. uglyGetMaxPage ships with a comment that calls it ugly and a reference to the proper implementation. That is better than dressing it up as intentional design — the next person reading the code knows exactly what to replace.

useRef for timers, not useState. Timer IDs are implementation details of cleanup logic. Storing them in state would schedule unnecessary re-renders on every keystroke. useRef keeps them stable across renders without triggering the component.
