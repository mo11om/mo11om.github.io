The Problem

Sending candidate resumes to a cloud LLM API creates two hard constraints that most hiring pipelines ignore: PII leaves your perimeter, and cost scales linearly with batch size. At 500 resumes per hiring cycle, even a compact GPT-4o prompt accumulates thousands of dollars in token spend — before accounting for revision loops.

The alternative is local inference. But local models produce unstructured output, hallucinate missing skills, and have no built-in retry logic. The engineering challenge is building a pipeline that runs entirely on-prem, produces reliable structured JSON, and self-corrects when the model lies.

LuminaLocal solves this with a three-node LangGraph agent, a multi-provider LLM factory, and a critique loop that catches hallucinations before they reach the CSV export.

System Design

Multi-Provider LLM Abstraction

The core abstraction is LLMFactory — a static factory that returns a unified BaseChatModel interface regardless of the underlying provider. This matters because Ollama, OpenAI, and vLLM all have different API shapes, JSON mode flags, and base URL conventions.

ProviderRetrieval StrategyJSON Mode
OllamaRAG (ChromaDB vector search)format: "json"
OpenAIFull resume text (RAG bypassed)response_format: json_object
vLLMRAG (ChromaDB vector search)response_format: json_object

The RAG toggle is the most consequential design decision: for local models, chunking and vector retrieval reduce context length and latency. For cloud providers, the full resume text is cheaper than an embedding round-trip at query time — so RAG is disabled entirely.

LangGraph Three-Node Pipeline

The graph has three nodes wired with a conditional feedback edge:

Retrieve → Analyze → Critique
               ↑         |
               └─ FAIL ──┘  (max 3 revisions)
  • Retrieve: conditional — RAG retriever or full text pass-through
  • Analyze: produces structured AnalysisResult JSON with classification, missing skills, and a two-sentence summary
  • Critique: audits the analysis against the resume context; outputs PASS or FAIL: <reason>

State is a TypedDict threaded through every node, accumulating revision_count and token_usage across iterations.

Self-Correction Loop

The critique node uses a separate LLM call (without JSON mode) to audit the analysis. It checks two failure modes:

  1. False negative: analysis claims a skill is missing, but the skill appears in the resume context
  2. Hallucination: analysis claims a skill is present, but the resume has no evidence of it

A FAIL verdict injects the critique feedback into the next analyze call’s system prompt as ### CRITIQUE FROM PREVIOUS ATTEMPT. The loop terminates at three revisions regardless of outcome, preventing infinite retry cycles.

Prompt Cache Optimization

The analyze node’s system prompt is structured specifically to maximize cache hits on providers that support prompt caching:

  1. Static resume context — placed first, never changes between jobs
  2. Static classification rules — placed immediately after
  3. Dynamic job info and feedback — placed last, only this portion changes per query

This order means the first two sections are cacheable across an entire batch run, reducing effective token cost for cloud providers.

class LLMFactory:
    @staticmethod
    def get_llm(provider: str, model_name: str, json_mode: bool = False, **kwargs):
        provider = provider.lower()
        temperature = kwargs.get("temperature", 0.1)

        if provider == "openai":
            model_kwargs = {"response_format": {"type": "json_object"}} if json_mode else {}
            return ChatOpenAI(model=model_name, temperature=temperature, model_kwargs=model_kwargs)

        elif provider == "ollama":
            base_url = kwargs.get("base_url", "http://localhost:11434")
            return ChatOllama(
                model=model_name, temperature=temperature, base_url=base_url,
                **({"format": "json"} if json_mode else {})
            )

        elif provider == "vllm":
            base_url = kwargs.get("base_url", "http://localhost:8000/v1")
            model_kwargs = {"response_format": {"type": "json_object"}} if json_mode else {}
            return ChatOpenAI(
                model=model_name, temperature=temperature,
                openai_api_base=base_url, openai_api_key=os.getenv("VLLM_API_KEY", "EMPTY"),
                model_kwargs=model_kwargs
            )

def build(self):
    workflow = StateGraph(AgenticState)
    workflow.add_node("retrieve", self.retrieve_node)
    workflow.add_node("analyze", self.analyze_node)
    workflow.add_node("critique", self.critique_node)

    workflow.set_entry_point("retrieve")
    workflow.add_edge("retrieve", "analyze")
    workflow.add_edge("analyze", "critique")
    workflow.add_conditional_edges(
        "critique",
        self.route_to_revision,
        {"analyze": "analyze", END: END}
    )
    return workflow.compile()

def route_to_revision(self, state: AgenticState) -> str:
    if state.get("revision_count", 0) >= 3:
        return END
    if "FAIL" in state.get("feedback", ""):
        print("Hallucination detected! Routing back to Analyze Node...")
        return "analyze"
    return END

Takeaway

Local inference is not free. The first batch run pays a cold-start cost: embedding the resume into ChromaDB takes 10–30 seconds depending on model size. Subsequent runs reuse the persisted vector store. Cloud inference skips this entirely but pays per token — the break-even point is roughly 20–30 jobs per resume before local inference wins on cost.

Pydantic catches silent failures. Without schema validation, a malformed LLM response silently corrupts the CSV output. The AnalysisResult model and the regex JSON extraction (re.search(r'\{.*\}', response_text, re.DOTALL)) together create two defense layers: structured output coercion and fallback extraction.

The critique loop has diminishing returns. In practice, most analyses pass on the first critique. The three-revision cap exists to bound worst-case token spend on cloud providers — the token_usage_report.csv export makes this cost visible per job title.

The logical next step is a response cache keyed on (resume_hash, job_hash) — identical re-screenings should return instantly without a new LLM call.
