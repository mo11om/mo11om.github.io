RTX 5090 Ubuntu Server

How to provision a multi-drive Ubuntu Server for GPU workloads using dual RAID 10 arrays and btrfs compression, while navigating RTX 5090 driver pitfalls.

The Problem

Standing up a GPU compute server sounds straightforward until the RTX 5090 ships with kernel modules that aren’t yet merged upstream. You boot Ubuntu Server, run nvidia-smi, and get nothing — no device, no driver version, just silence. Meanwhile the storage layer has its own complexity: you want OS resilience on a fast ext4 RAID 10, but service data benefits from transparent compression that ext4 can’t offer. Two filesystems, two RAID arrays, one installation pass, and a GPU that may not cooperate until you know exactly where to look in dmesg.

The non-trivial part isn’t any single step — it’s that each layer (disk partitioning, RAID assembly, fstab, and driver loading) has silent failure modes that only surface later, sometimes after a reboot when recovery is harder.

System Design

Storage Architecture

The server uses two software RAID 10 arrays built from the same set of physical drives. Each drive is partitioned into two regions during the Ubuntu installer’s guided partitioning step, so both arrays span all disks simultaneously.

ArrayFilesystemMount PointPurpose
RAID 10 (2 TB × N)ext4/OS, system packages, swap
RAID 10 (remaining)btrfs/serviceContainers, model weights, datasets

Why two arrays instead of one? ext4 has lower per-operation overhead for small random writes (package installs, journal writes). btrfs zstd compression typically achieves 1.5–2× reduction on model checkpoints and log files with negligible CPU cost at the default level. Mixing the two filesystems on separate RAID arrays avoids btrfs mount-option entanglement with the root device and lets each array tune independently.

Trade-off: managing two md devices adds operational surface area. mdadm must be kept synchronized across both arrays after any drive replacement, and btrfs scrub schedules are separate from ext4 fsck.

GPU Driver Pipeline

The RTX 5090 falls into a catch-22 on a headless server: the Ubuntu installer doesn’t ship the required proprietary module, and ubuntu-drivers autoinstall may pull a version that doesn’t yet support the Blackwell architecture. The diagnostic path starts at dmesg — the kernel will log whether it found the PCIe device but failed to bind a driver, or whether the device is invisible entirely.

Network access is a hard dependency for driver installation. The 5090’s required kernel modules aren’t in the base ISO, so if the built-in NIC lacks an in-tree driver, you need either Wi-Fi or a PCIe Ethernet card with known kernel support before the OS install completes.

Implementation

Dual-array fstab with btrfs mount options:

# /etc/fstab — btrfs entry after adding compression flags
# Before (installer default):
/dev/disk/by-id/md-uuid-d0bc07c8:8c200c66:0315fa2e:b7fe7076-part1 /service btrfs defaults 0 1

# After (add compress and noatime; no trailing comma):
/dev/disk/by-id/md-uuid-d0bc07c8:8c200c66:0315fa2e:b7fe7076-part1 /service btrfs defaults,compress=zstd,noatime 0 1

noatime prevents btrfs from writing an access timestamp on every read, which matters on a dataset volume with millions of small files. Omitting it causes visible write amplification under model training workloads.

Remounting without a reboot:

sudo umount /service
sudo mount -t btrfs -o defaults,compress=zstd,noatime /dev/md1p1 /service
mount | grep /service          # confirm options are live
sudo systemctl daemon-reload   # required if systemd cached the old fstab

The daemon-reload step is easy to skip — systemd will warn you with “your fstab has been modified, but systemd still uses the old version” if you don’t run it. Skipping it means the next systemctl mount command silently uses stale options.

# First look: did the kernel see the GPU at all?
dmesg | grep -i nvidia

# Common failure signatures:
# "nouveau: DRM: unknown chipset"  → nouveau loaded instead of proprietary driver
# silence on the nvidia line       → PCI device not recognized (too new for this kernel)
# "NVRM: GPU-00000000:01:00.0"     → device found but module load failed

The log output maps directly to three different remediation paths: blacklist nouveau, update the kernel, or manually install a pre-release driver package. Knowing which one applies before searching saves significant time.

Takeaway

Partition before you RAID. The Ubuntu installer’s software RAID workflow requires unformatted partitions as inputs, not whole disks. Skipping the explicit partition step and pointing RAID at raw devices works initially but makes array expansion and per-partition options impossible later.

fstab options are not applied retroactively. Adding compress=zstd to an existing btrfs mount compresses only new writes. Data already on disk stays uncompressed until a btrfs filesystem defrag -r -czstd /service pass, which is I/O-intensive. Plan the compression strategy before the volume fills.

Driver debugging starts at dmesg, not Google. The RTX 5090’s driver situation changes monthly as Blackwell support matures. The dmesg | grep -i nvidia output gives a version-specific error that narrows the fix immediately; generic forum advice often targets a different kernel or driver generation.

The current setup lacks automated RAID health monitoring — mdadm --monitor should be configured to send alerts on degraded arrays. The next step is adding a systemd timer for btrfs scrub on /service and wiring both alert streams into a notification channel before putting the server under load.
