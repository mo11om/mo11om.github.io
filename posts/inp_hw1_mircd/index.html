The Problem

The naive approach to a multi-client server is one thread per connection: accept a client, spawn a thread, read in a loop. This works up to a few hundred clients, then collapses under thread scheduling overhead and per-thread stack allocation.

The alternative is I/O multiplexing: one thread, one poll() call, many file descriptors. The operating system tells you which sockets are ready to read. You service them in sequence. No locks, no context switches between threads, no stack overhead per client.

This IRC server supports 1024 simultaneous connections in a single thread and roughly 775 lines of C++. The constraint that makes this tractable: IRC is a protocol designed around short bursts of text. Clients spend most of their time idle. poll() excels at exactly this workload.

System Design

The poll() Event Loop

The server allocates a fixed array of struct pollfd client[OPEN_MAX] (1024 entries). client[0] holds the listening socket. Entries 1–1023 hold connected client sockets, or -1 if the slot is free.

Each iteration of the main loop calls poll(client, maxi+1, INFTIM) — blocking indefinitely until at least one fd becomes readable. INFTIM (-1) means no timeout: the server sleeps with zero CPU until there is real work to do.

On wakeup, the loop checks two cases:

  1. New connection (client[0].revents & POLLRDNORM): accept(), find a free slot, register the fd, call set_map() to record the client’s IP and assign a random nickname.
  2. Client data (any client[i].revents & POLLRDNORM): read() up to 500 bytes, dispatch to deal_input(), or clean up on zero-byte read (connection closed).

State Machine: Three Maps

All IRC session state lives in three data structures:

When a client disconnects, erase_server() iterates map_fd_channels[fd] and removes the client from every channel’s member map, then erases the fd from the global registry. This avoids dangling references without reference counting.

Protocol Parsing

IRC messages are newline-delimited (\r\n). Each read() call can contain multiple commands. The parser splits on \r\n first (split_instructions()), then tokenizes each command by whitespace (simple_tokenizer()).

Command dispatch is a flat if/else if chain over arg[0]:

NICK → update nickname in map
USER → register client in server channel, send welcome
JOIN → add_channel(), broadcast join to channel members
PRIVMSG → privmsg() → write to each member fd
QUIT → erase_server(), close fd
...
421 Unknown command → for anything else

Numeric reply codes follow RFC 1459: 001 for welcome, 321/322/323 for channel list, 353/366 for names, 421 for unknown command.

Broadcast Mechanics

privmsg() iterates the target channel’s map_channel_fd_nickname and calls write() synchronously to each member’s fd. This is head-of-line blocking: if one client’s socket buffer is full (the client is slow), write() blocks, and every other member waits.

For a server where all clients are on a LAN and sockets rarely fill, this is acceptable. For a server with heterogeneous client speeds, the fix is a per-client outbound queue and non-blocking writes with O_NONBLOCK — significantly more complex.

client[0].fd = listenfd;
client[0].events = POLLRDNORM;
for (i = 1; i < OPEN_MAX; i++) client[i].fd = -1;

for (;;) {
    nready = poll(client, maxi + 1, INFTIM);

    if (client[0].revents & POLLRDNORM) {
        connfd = accept(listenfd, (SA*)&cliaddr, &clilen);
        for (i = 1; i < OPEN_MAX; i++)
            if (client[i].fd < 0) { client[i].fd = connfd; break; }
        client[i].events = POLLRDNORM;
        if (i > maxi) maxi = i;
        set_map(connfd);
        if (--nready <= 0) continue;
    }

    for (i = 1; i <= maxi; i++) {
        if ((sockfd = client[i].fd) < 0) continue;
        if (client[i].revents & (POLLRDNORM | POLL_ERR)) {
            if ((n = read(sockfd, buf, MAXLINE)) == 0) {
                erase_map(sockfd);
                close(sockfd);
                client[i].fd = -1;
            } else {
                deal_input(sockfd, buf, n);
            }
            if (--nready <= 0) break;
        }
    }
}

int add_channel(int sockfd, string nickname, string channel_name) {
    int i;
    for (i = 1; i < all_channels.size(); i++) {
        if (all_channels.at(i)->name == channel_name) {
            all_channels.at(i)->map_channel_fd_nickname[sockfd] = nickname;
            map_fd_channels[sockfd].push_back(i);
            return i;
        }
    }
    // channel doesn't exist — create it
    channel* tmp = new channel;
    tmp->name = channel_name;
    tmp->map_channel_fd_nickname[sockfd] = nickname;
    all_channels.push_back(tmp);
    map_fd_channels[sockfd].push_back(i);
    return i;
}

void erase_server(int fd) {
    auto iter = map_fd_channels.find(fd);
    for (auto ch = iter->second.begin(); ch != iter->second.end(); ch++) {
        all_channels[*ch]->map_channel_fd_nickname.erase(fd);
    }
    map_fd_channels.erase(fd);
}

Takeaway

poll() scales — until broadcast does not. The event loop itself handles 1024 idle clients cheaply. The bottleneck appears in privmsg(): synchronous writes to every channel member serialize delivery behind the slowest socket. A production IRC server (ircd, ngircd) maintains a per-client write buffer and uses non-blocking I/O to avoid this entirely.

new channel without delete leaks memory. Every add_channel() call that creates a new channel allocates a channel* on the heap. When a channel empties, the pointer remains in all_channels and the memory is never freed. For a long-running server with churn, this accumulates. A shared_ptr<channel> with cleanup on empty membership would fix it without requiring manual delete.

Single-threaded is also simpler to reason about. There are no mutexes, no race conditions, no need to think about which thread owns which data structure. This is the core appeal of the event-loop model — and why Node.js, Redis, and Nginx all use it. The complexity cost is paid later, when you need to parallelize CPU-bound work or handle slow writes without blocking the loop.
